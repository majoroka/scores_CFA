- publique resultados frescos rapidamente quando a FPF já os expõe;
- reflita alterações de data/hora/local antes dos jogos;
- distinga atraso normal da FPF, bloqueio técnico e falha de parser;
- torne os workflows “verdes” realmente significativos.

INVESTIGACAO DIRIGIDA ATUAL
Casos de referência:
- seniores
- infantis-b

Factos já provados:
1. a app/Pages está a servir JSON stale nestas duas competições;
2. a origem FPF já tem dados mais frescos;
3. os fetchers locais conseguem gerar JSON novo;
4. num run verde real, os fetchers foram executados mas ficaram classificados como `unchanged`;
5. o commit automático desse run alterou essencialmente agregados, não os JSON destas competições.

Conclusão provisória:
- o bloqueio atual está entre:
  - deteção de mudança útil;
  - persistência do payload atualizado;
  - decisão de commit/publicação;
- não está no frontend;
- não está no deploy final do Pages;
- e, para estes 2 casos, não está no parser base nem no acesso HTTP real à FPF.

CHECKLIST OPERACIONAL ATUAL
Objetivo imediato:
- fechar a causa exata de `source fresco -> fetch executa -> workflow decide unchanged -> Pages continua stale`

Passos em curso antes do Bloco E:
1. Confirmar em cada run dirigido:
   - `source_changed`
   - `parsed_changed`
   - `published_changed`
2. Guardar como artifact obrigatório:
   - `cache/fetch_plan.json`
   - `cache/fetch_report.json`
   - `cache/sync_metadata/*.json`
   - `cache/errors/**`
   - snapshots HTML de fixtures corridos
3. Verificar por fixture:
   - se a origem mudou;
   - se o parser detetou a mudança;
   - se o round final mudou;
   - se o ficheiro final mudou no `git diff`;
   - se o commit automático incluiu essa mudança
4. Falhar ou degradar explicitamente o run quando:
   - `source_changed = true`
   - e `published_changed = false`
5. Só depois avançar para Bloco E
